2. Read the source code at the reported location.
3. Find every caller of the reported function or code construct in the codebase.

5. Trace the realistic network/input path to the reported code. Can an unauthenticated

Before rendering your verdict, run this slop-loophole checklist. Each item is a known
failure mode from prior runs. Check each one explicitly:

SL-1 SYNTHETIC TRIGGER
Does the trigger call the vulnerable function directly with fabricated arguments, bypassing
the real callers? grep every caller of the reported function. If all real callers pass
compile-time constants (sizeof, hardcoded values) at the reported vulnerable parameter,
and the trigger constructs a value the real callers never would, the finding is invalid.
The trigger must reflect an attack path a real caller or real network input can produce.

SL-2 SANITIZER-ONLY UB
Does the finding rely solely on UB that sanitizers flag but that compilers handle
predictably on all realistic platforms (x86/ARM, GCC/Clang, -O1/-O2)? Common examples:
signed integer overflow of the "expected" two's-complement form, left-shifting into the
sign bit when the result is immediately used as unsigned. If the runtime behavior is
identical with or without the UB, and no attacker-observable consequence follows
(wrong value, incorrect control flow, exploitable memory state), reject as SL-2.
Note: SL-2 is a code-quality issue, not a security finding.

SL-3 DEAD-CODE GATE
Is the vulnerable code compiled into the default build? Check both:
(a) Source-level: is the code inside an #ifdef / #if guard? If so, grep the project's
    config headers (config.h, configure output, CMake-generated defines, etc.) to
    determine whether the guard macro is defined in a default build. Commented-out
    #defines, or macros only set by optional ./configure flags, mean the code is dead.
(b) Object-level: find the compiled object file for the source (e.g. foo.c → foo.o)
    and run `nm -g --defined-only <path>/foo.o | grep <function_name>`. If the
    vulnerable function does not appear, it was preprocessed out.
(c) Cross-package: if any cited caller lives in a different directory or sibling
    package from the focus file (e.g. focus is in miniupnpd/ but caller is cited
    in minissdpd/ or miniupnpc-libuv/), verify that package was built at all.
    Run `ls <caller_dir>/<caller_file>.o`. If no .o exists, that package was not
    compiled in the default build and the entire caller path is dead — regardless
    of whether the code is syntactically reachable.
If any check shows the code is not compiled in the default configuration, the
finding is INTRACTABLE with reason "dead code — package <dir> not built".
A bug in uncompiled code has no security impact.
